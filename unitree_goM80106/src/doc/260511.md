# 2026-05-11 修改记录：Ctrl+C 退出时自动停止电机

## 问题

`example_goM8010_6_motor.cpp` 中有一个 `while(true)` 无限循环持续通过串口向电机发送速度指令（`dq = -6.28 * gearRatio`）。直接 Ctrl+C 杀掉进程时，**电机收不到停止指令**，会继续维持最后收到的速度目标运行。

## 第一次修改（错误）

思路：注册 SIGINT 信号处理函数，在里面发停止指令。

```cpp
static std::atomic<bool> running{true};
static SerialPort* global_serial = nullptr;
static MotorCmd* global_cmd = nullptr;

void stopMotor(int) {
  running = false;
  if (global_serial && global_cmd) {
    global_cmd->dq = 0.0f;
    // ... 清零参数 ...
    global_serial->sendRecv(global_cmd, nullptr);  // ← 问题在这里
  }
  exit(0);
}

int main() {
  // ...
  global_serial = &serial;
  global_cmd = &cmd;
  std::signal(SIGINT, stopMotor);
  // ...
}
```

### 错误原因

信号处理函数中调用 `sendRecv`（串口 I/O）**不安全**。信号处理函数运行时，程序可能正处于 `sendRecv` 内部的 mutex 加锁状态，信号处理函数再次调用 `sendRecv` 会导致：

- **死锁**：同一个线程重复锁住非递归互斥锁
- **数据损坏**：信号上下文破坏了正在进行的串口读写状态

C++ 标准规定：信号处理函数只能调用 async-signal-safe 的函数（如 `write`、`exit`），不能做 I/O、不能锁 mutex、不能分配堆内存。

## 第二次修改（正确）

### 原则

> 信号处理函数只做一件事：设标志位。I/O 和清理回正常上下文中做。

```cpp
static volatile sig_atomic_t running = 1;

void signalHandler(int) {
  running = 0;   // 唯一动作：设标志位
}

int main() {
  std::signal(SIGINT, signalHandler);

  while(running) {
    // ... 正常发送控制指令 ...
    serial.sendRecv(&cmd, &data);
    usleep(200);
  }

  // while 循环自然退出后，在正常代码路径中发送停止指令
  cmd.dq  = 0.0f;
  cmd.q   = 0.0f;
  cmd.tau = 0.0f;
  cmd.kp  = 0.0f;
  cmd.kd  = 0.0f;
  serial.sendRecv(&cmd, &data);   // 安全：不在信号上下文中

  return 0;
}
```

### 两个关键要点

| 要点 | 说明 |
|------|------|
| **信号处理函数不做 I/O** | `running = 0` 是唯一的安全操作，其余事情回到 `main()` 的正常路径执行 |
| `volatile sig_atomic_t` | C/C++ 标准为信号处理场景提供的原子类型，保证在信号上下文的读写不被编译器优化撕裂 |

### 退出流程

```
Ctrl+C → SIGINT → signalHandler() 设 running = 0 → 返回被中断处
→ usleep() return（或下一轮循环检查 running）→ while 退出
→ 发送停止指令给电机 → return 0
```

## LocationMode 检查

文件 `LocationMode_8010t1.cpp` 是位置控制模式，使用 `kp=0.15, kd=0.1` 控制电机到达目标位置。

### 正确之处

| 项目 | 说明 |
|------|------|
| 停机流程 | 信号处理设标志 → 主循环退出 → 发停止指令，模式正确 ✅ |
| 启动预读 | 先发一次指令读当前 `data.q` 作为起始位置，避免从 0 突跳 ✅ |
| 位置缓变 | `stepSize = 0.5 rad` 逐步逼近目标，不会一步到位产生大力矩 ✅ |

### 问题

**1. `mode = 0` 急停更突兀**

位置模式下电机正用 `kp` 维持一个保持力矩，切 `mode = 0` 等于突然松开这个力矩。如果机构有重力负载（如悬空关节），电机会瞬间弹动。与速度模式的区别：

| 模式 | 切 mode 0 的效果 |
|------|-----------------|
| 速度控制（`kp=0, dq≠0`） | 高速行驶拉手刹 |
| 位置控制（`kp>0, q=目标`） | 拉紧的弹簧突然松手 |

**2. 注释错误**

`usleep(200);  // 200ms 周期` ← `usleep` 单位是微秒，实际为 0.2 ms。配合 `stepSize = 0.5 rad`，位置逼近过程实际只有约 0.37 秒，远快于注释暗示的节奏。

### 停机手法对比

| 方式 | 代码 | 效果 |
|------|------|------|
| 仅清参数（FOC 模式） | `dq=0, kp=0, kd=0`，不设 mode | 自然减速/卸力，平滑但滑行距离长 |
| 直接切 mode 0 | `cmd.mode = 0` | 急停，生硬 |
| 逐步卸力 → 再切 mode 0 | 循环递减 `kp`/`kd`，最后 `mode=0` | 先平滑卸力再停止，推荐 |

### 推荐做法

位置控制下退出时，先逐步撤销力矩再制动：

```cpp
  // 第1步：渐进撤力（保持位置目标不变，逐步降低 kp，加大 kd 吸收动能）
  for (int i = 0; i < 100; i++) {
    float t = (100 - i) / 100.0f;
    cmd.kp = 0.15f * t;
    cmd.kd = 0.1f * (1.0f + t);    // 阻尼逐渐增大
    serial.sendRecv(&cmd, &data);
    usleep(200);
  }

  // 第2步：力矩已接近 0，切停止模式
  cmd.mode = 0;
  cmd.q = 0; cmd.dq = 0; cmd.tau = 0;
  cmd.kp = 0; cmd.kd = 0;
  serial.sendRecv(&cmd, &data);
```

### 通用原则

| 场景 | 做法 |
|------|------|
| 速度控制 | 清 `dq=0` 自然减速即可，无需切 mode |
| 位置控制 | 先卸力矩（递减 kp），再切 mode 0 |
| 急停/抱闸 | 直接 `mode=0`（如掉电保护、紧急停止） |
